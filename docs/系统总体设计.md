# 系统总体设计

## 1. 设计目标

本系统面向高校教学反馈业务，设计目标如下：

1. 结构清晰，便于毕业设计答辩展示
2. 模块解耦，便于后续逐步实现
3. 业务闭环完整，覆盖反馈、处理、回复、统计
4. 支持后续扩展数据库、鉴权、大模型能力

## 2. 技术路线

考虑当前环境与毕业设计可落地性，项目采用以下方案：

### 2.1 当前实现方案

1. 后端：Java 11 + Spring Boot 2.7 + Spring JDBC
2. 前端：Vue 3 + TypeScript + Vite + Element Plus 管理端
3. 通信方式：HTTP + JSON
4. 数据库：MySQL 8
5. 鉴权：本地 token，预留学校统一身份认证接入空间
6. 文档层：使用 Markdown 维护需求、设计与接口契约

## 3. 架构分层

项目采用前后端分离架构：

1. 前端应用负责界面渲染、交互、路由状态和接口调用
2. 后端应用负责 API、业务规则、数据组织和权限边界
3. 前后端通过固定 API 前缀 `/api/v1` 通信
4. 后端开启 CORS，支持本地开发联调

后端采用 Spring Boot 分层结构：

1. `api`：接口控制器
2. `service`：业务服务与数据访问封装
3. `security`：登录、密码校验与 token 处理
4. `config`：跨域和应用配置
5. `common`：统一响应结构

每个业务模块推荐包含：

1. `controller`：处理 HTTP 请求
2. `service`：承载业务规则
3. `repository`：封装数据访问
4. `mapper`：视图对象转换

## 5. 目录规范

```text
backend-spring/
├─ pom.xml
└─ src/main/
   ├─ java/com/sdust/feedback/
   │  ├─ api/
   │  ├─ common/
   │  ├─ config/
   │  ├─ security/
   │  ├─ service/
   │  └─ StudentFeedbackApplication.java
   └─ resources/application.yml

frontend/
├─ index.html
├─ vite.config.js
├─ styles/
└─ src/
   ├─ api/
   ├─ shared/
   ├─ styles/
   ├─ App.vue
   ├─ main.ts
   └─ config.ts
```

约束如下：

1. 控制器不直接写业务规则
2. 服务层不直接依赖 HTTP 细节
3. 仓储层只关心数据访问
4. 公共逻辑沉淀到 `shared`
5. 配置项统一放到 `config`

## 6. API 规划

当前先定义基础接口：

1. `GET /api/v1/health`：服务健康检查
2. `GET /api/v1/meta/modules`：模块总览
3. `GET /api/v1/auth/me`：当前用户
4. `POST /api/v1/auth/login`：模拟登录
5. `GET /api/v1/dashboard/summary`：首页统计
6. `GET /api/v1/users`：用户列表
7. `GET /api/v1/master-data/:resource`：基础数据列表
8. `POST /api/v1/master-data/:resource`：新增基础数据
9. `GET /api/v1/schedules/weekly-tasks`：周反馈任务列表
10. `GET /api/v1/schedules/weekly-task-compliance`：学委履职标记
11. `GET /api/v1/feedbacks/weekly`：周反馈列表
12. `GET /api/v1/feedbacks/realtime`：实时反馈列表
13. `POST /api/v1/feedbacks/realtime`：新增实时反馈
14. `POST /api/v1/feedbacks/reply`：管理员处理意见
15. `GET /api/v1/feedbacks/replies`：反馈处理记录
16. `PATCH /api/v1/feedbacks/status`：反馈状态流转

## 7. 数据持久化策略

当前后端直接连接 MySQL，数据库连接配置位于：

```text
backend-spring/src/main/resources/application.yml
```

核心数据包括用户、角色、院系、专业、班级、教师、课程、学期、教学任务、周反馈任务、周反馈、实时反馈、管理员回复、敏感词和敏感标记。

2. 操作日志

4. AI 总评生成
5. 与学校统一认证、课表接口联调

## 8. 代码维护原则

1. 单一职责：一个文件只解决一个层面的事
2. 模块隔离：不同业务模块尽量不直接互相访问内部实现
3. 命名统一：后端使用 `Controller`、`Service`、`Config`、`Security` 等 Java 类后缀
4. 先定义结构，再补细节，避免后续推翻重写
5. 能被论文解释清楚的设计优先于过度复杂的框架
