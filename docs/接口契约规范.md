# 接口契约规范

本文档用于约束前端 TypeScript 类型和后端 Spring Boot 返回结构，避免后续业务扩展时出现字段不一致、状态值混乱、权限边界不清的问题。

## 统一响应结构

所有业务接口统一返回：

```json
{
  "success": true,
  "message": "操作成功",
  "data": {}
}
```

失败时：

```json
{
  "success": false,
  "message": "错误原因",
  "data": null
}
```

前端对应类型位于：

```text
frontend/src/api/types.ts
```

后端对应结构位于：

```text
backend-spring/src/main/java/com/sdust/feedback/common/ApiResponse.java
```

## 登录认证

当前阶段采用本地 JWT 风格 token，后续接入学校统一身份认证时，前端 token 保存逻辑不需要大改，只需要替换后端登录实现。

接口：

```text
POST /api/v1/auth/login
GET  /api/v1/auth/me
```

前端请求头：

```text
Authorization: Bearer <token>
```

## 角色约定

```text
SUPER_ADMIN              超级管理员
DEPARTMENT_ADMIN         院系管理员
CLASS_REPRESENTATIVE     学委
STUDENT                  普通学生
```

后续权限隔离规则：

1. 超级管理员可以查看全部院系、班级、课程、反馈。
2. 院系管理员只能查看本院系及其下属专业、班级、课程、反馈。
3. 学委必须按周提交本班课程反馈。
4. 普通学生可以提交实时反馈并查看本人反馈。

## 状态值约定

周反馈任务：

```text
PENDING        待提交
IN_PROGRESS    进行中
CLOSED         已关闭
```

实时反馈/周反馈处理状态：

```text
SUBMITTED       已提交
PENDING_REPLY   待回复
IN_PROGRESS     已受理/处理中
REPLIED         已回复
CLOSED          已关闭
```

反馈处理闭环：

```text
GET   /api/v1/feedbacks/replies?feedbackType=REALTIME&feedbackId=1
POST  /api/v1/feedbacks/reply
PATCH /api/v1/feedbacks/status
```

处理规则：

```text
1. 只有 SUPER_ADMIN 和 DEPARTMENT_ADMIN 可以回复或更新反馈处理状态。
2. 院系管理员只能处理本院系数据。
3. 学生可以查看自己反馈对应的回复记录。
4. 回复内容会写入 feedback_reply 表，处理状态同步更新到周反馈或实时反馈主表。
```

紧急程度：

```text
HIGH
MEDIUM
LOW
```

学委履职状态：

```text
PENDING            未到截止时间，暂未提交
OVERDUE_MISSING    已过截止时间，仍未提交
LATE_SUBMITTED     已提交，但晚于截止时间
SUBMITTED          已按时提交
```

## 权限过滤规则

当前后端列表类接口已经根据登录用户自动做数据范围过滤：

```text
SUPER_ADMIN              查看全部数据
DEPARTMENT_ADMIN         仅查看本院系数据
CLASS_REPRESENTATIVE     仅查看本人/本班相关数据
STUDENT                  仅查看本人/本班相关数据
```

过滤覆盖范围：

```text
GET /api/v1/users
GET /api/v1/master-data/{resource}
GET /api/v1/schedules/weekly-tasks
GET /api/v1/schedules/weekly-task-compliance
GET /api/v1/feedbacks/weekly
GET /api/v1/feedbacks/realtime
GET /api/v1/feedbacks/replies
GET /api/v1/feedbacks/flags
GET /api/v1/dashboard/summary
```

## 下一步接口规范化方向

1. 为新增、编辑、删除接口补充 DTO 校验。
2. 将 `Map<String, Object>` 逐步替换为明确的 Request/Response 类。
3. 增加分页参数：`pageNo`、`pageSize`、`keyword`。
4. 增加操作日志接口，用于记录管理员回复、任务生成、课表导入等关键操作。
