# 学生反馈系统毕业设计需求梳理

## 1. 当前题目方向

建议题目：

`基于 Web 的高校学生教学反馈与紧急事件上报系统的设计与实现`

也可扩展为：

`融合大模型文本汇总的高校教学反馈管理系统设计与实现`

## 2. 需求来源

本需求文档依据以下内容整理：

1. 你的文字笔记
2. 老师口述对话全文
3. 示例文件 `2024-2025-2第13周外教课程反馈--学生层面-demo.xls`
4. 示例文件 `2024-2025-2第13周外教课程反馈--教师层面-demo.xlsx`

## 3. 项目核心目标

建设一个面向高校教学场景的反馈系统，支持：

1. 学委按周对本周全部课程进行教学反馈
2. 普通学生随时提交一般反馈或紧急反馈
3. 管理员分级查看、处理、回复、统计反馈信息
4. 对课程、班级、教师、院系维度进行汇总分析
5. 对文本反馈进行敏感词识别、重点标记和总评生成
6. 与课表数据对接，按学年学期、周次、课程维度自动分发待反馈任务

## 4. 从老师对话中提取的完整业务要求

### 4.1 用户角色

系统至少包含以下角色：

1. 超级管理员
2. 院系管理员
3. 学委
4. 普通学生

### 4.2 学生侧要求

#### 学委

1. 学委必须反馈，不是可选动作
2. 学委需要按周反馈
3. 每周需要对当周全部课程信息进行评价
4. 反馈时间由系统管理员统一设置，不由学委决定
5. 系统需要在规定时间提醒学委进行反馈
6. 学委若未在规定时间内反馈，需要被系统记录
7. 未按时反馈需要形成“标记”，后续可用于奖惩
8. 管理端需要统计学委反馈次数
9. 管理端需要评估学委反馈质量
10. 可设置最低字数要求，避免“全都很好”这类敷衍反馈
11. 学委可查看自己提交的反馈
12. 学委端需要支持增删查改
13. 学委端最好支持统计查看

#### 普通学生

1. 普通学生也可以登录系统
2. 普通学生不局限于每周反馈
3. 普通学生可以随时反馈
4. 反馈内容不仅限于教学，也可以是与学习相关的沟通问题
5. 普通学生端支持增删查改
6. 普通学生侧重“随时反馈”

### 4.3 反馈类型

系统至少应支持两大类反馈：

1. 周反馈
2. 实时反馈/紧急反馈

#### 周反馈

1. 面向学委为主
2. 针对“当周上过的全部课程”
3. 需要结合课表自动生成待反馈课程列表
4. 重点反映课程开展情况、教学效果、问题建议

#### 实时反馈/紧急反馈

老师明确提到的场景包括：

1. 教室硬件问题
2. 多媒体设备不能正常运转
3. 老师无故离场
4. 老师早退
5. 上课过程中出现让学生不适的口述内容
6. 其他需要当天或随时上报的问题

### 4.4 管理侧要求

#### 超级管理员

1. 拥有最高权限
2. 负责用户基础信息管理
3. 支持学生信息导入
4. 支持教师信息新增
5. 支持用户信息增删查改
6. 支持课程信息增删查改
7. 负责课表对接
8. 课表应尽量实时抓取，而不是只做一次性录入
9. 需要维护学年学期
10. 需要维护周次
11. 需要在“大库”中进行查询统计
12. 能查看所有院系、课程、班级的情况
13. 需要对紧急处理情况给予反馈
14. 能统计学委反馈次数与质量

#### 院系管理员

1. 只能看到自己院系的数据
2. 只能查看自己院系学生信息与反馈信息
3. 能处理本院系反馈
4. 能查看课程评价、班级评价和课程总评

### 4.5 课表与任务分发

老师对这一部分要求非常明确：

1. 系统需要对接课表
2. 课表应尽量实时抓取相关数据
3. 每个学期都要有课表数据
4. 学年学期必须单独管理
5. 需要有周次概念
6. 每周系统应按学生身份、班级、当周课程自动生成待反馈课程
7. 学委反馈的是“这一周上过的课程”，不是某一天的课程
8. 分发逻辑要能根据学生所属班级和当周课程进行匹配

### 4.6 查询统计与汇总分析

1. 管理员可查看每个班对某门课程的评价
2. 管理员可查看某门课程的总评
3. 管理员可查看分评与总评
4. 可按课程、班级、教师、院系统计
5. 一个教师可能给多个班上同一门课，需要跨班汇总
6. 汇总不是简单 `join` 或拼接
7. 最好可以接入大模型，对多份文本反馈进行归纳润色并生成总评

### 4.7 文本反馈特点

老师明确强调：

1. 目前更偏向定性反馈，不是简单打分
2. 系统重点是“反映情况”
3. 需要知道发生了什么，而不是只给一个分数
4. 例如：
   - 外教语速过快
   - 知识点没有讲明确
   - 作业量过大
   - 水课等评价
5. 这些内容难以直接量化打分，更适合文本表达

### 4.8 敏感信息与质量监控

老师明确提出：

1. 反馈中的敏感词或敏感表述需要强标
2. 例如“水课”等词语需要重点标记
3. 系统需要帮助管理者精准定位问题课程
4. 这属于教学质量监控的一部分

### 4.9 回复与闭环

1. 紧急问题需要处理反馈
2. 学生提交反馈时，可自主选择“是否需要回复”
3. 若学生选择需要回复，管理员端应收到待回复事项
4. 管理员可以在系统内直接回复
5. 学生可以查看回复信息
6. 若管理员未及时回复，系统需要提醒
7. 形成“反馈 - 处理 - 回复 - 查看”的闭环

### 4.10 隐私与实名

从老师表述来看，目前倾向：

1. 默认实名
2. 暂时不一定需要匿名
3. 但可以考虑提供匿名或“统计时不见人，只见班级”的保护机制
4. 重点是对学生个人隐私的保护
5. 反馈数据不对外公开

## 9. 关键业务规则

1. 学委每周必须完成一次针对当周全部课程的反馈
2. 任务生成依赖课表、学年学期、周次、班级和学生身份
3. 普通学生可以不限时间提交反馈
4. 紧急反馈应允许即时上报
5. 反馈可以选择“是否需要回复”
6. 需要回复的反馈必须进入管理员待办
7. 院系管理员只能访问本院系数据
8. 超级管理员拥有全局可见权限
9. 系统应记录未按时反馈情况，供奖惩使用
10. 反馈内容可设置最少字数
11. 敏感文本要自动标记
12. 汇总分析以文本为核心，不以分数为核心

## 10. 论文和开题时可强调的创新点

你这个毕设很适合写下面几个创新点：

1. 将“周反馈 + 实时紧急反馈”结合，覆盖常规评价与突发事件上报
2. 与课表联动，自动生成周反馈任务，减少手工分发
3. 引入角色分级权限，实现超管、院系管理员、学委、普通学生的精细化管理
4. 引入文本敏感信息识别，提升教学质量监测能力
5. 借助大模型对多源文本反馈进行自动汇总，生成课程总评
6. 建立“提交 - 处理 - 回复 - 查看”的闭环反馈机制

## 11. 当前仍需你后续确认的点

下面这些老师给的是方向，但你后面做系统时需要进一步定稿：

1. 是否最终支持匿名反馈
2. 奖惩规则是仅记录标记，还是系统直接给出等级
3. 课表是通过接口实时抓取，还是先做 Excel 导入模拟
4. 反馈质量评价是人工评价、规则评价，还是大模型评价
5. 紧急反馈是否需要状态流转，如待处理、处理中、已解决、已回复
6. 普通学生可反馈的范围是否仅限教学问题
7. 是否需要消息通知，如站内信、短信、邮件或仅系统提醒

## 12. 最适合你的实现策略

如果你是做毕业设计，建议你把范围控制在“能做完、能演示、能写论文”：

1. 第一阶段先做本地导入课表，不强依赖真实教务系统接口
2. 先实现实名登录与角色权限
3. 先实现周反馈、实时反馈、回复闭环
4. 再做统计分析
5. 最后加一个大模型总评功能作为亮点

这样更稳，论文也更好写。
